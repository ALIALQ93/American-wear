# النشر على الإنترنت (بدون تشغيل على جهازك)

هذا الملف موجّه لمن **ليس مبرمجاً** لكنه يريد فكرة متجر إلكتروني، ويفكّر في **GitHub** كـ«استضافة».

---

## 1) ماذا يفعل GitHub فعلاً؟

| ما تفعله | هل يكفي لمتجرك بهذا المشروع؟ |
|----------|-------------------------------|
| **رفع الكود (Repository)** | نعم — مكان آمن لحفظ المشروع والنسخ. |
| **GitHub Actions** (مثل ملف `.github/workflows/ci.yml`) | يتحقق أن المشروع **يُبنى** عند الدفع — مفيد للجودة، **ليس** تشغيل المتجر للزبائن 24/7. |
| **GitHub Pages** | يستضيف **صفحات ثابتة** فقط (HTML/CSS/JS بعد `npm run build`). **لا** يشغّل خادم **Express** ولا يتصل بـ Postgres بنفس طريقة التطوير المحلي **إلا** إذا أعيد تصميم المشروع أو فُصلت الواجهة عن الخادم بتعديلات برمجية. |

**الخلاصة:** GitHub ممتاز **كمستودع كود ونسخ احتياطي**. **استضافة المتجر الحقيقية** (حتى يفتحه الناس من الرابط) تحتاج عادةً **خدمة استضافة أخرى** تشغّل Node، مع الإبقاء على **Supabase** لقاعدة البيانات والمصادقة كما هو الآن.

---

## 2) ماذا يحتاج هذا المشروع ليعمل على الإنترنت؟

1. **واجهة** (ملفات من `npm run build` داخل مجلد `dist/`).
2. **خادم API** (ملف `server/index.js` — Express على Node).
3. **PostgreSQL** (لديك عبر **Supabase** — يبقى كما هو).
4. **متغيرات البيئة** (نفس فكرة `.env` لكن تُضبط في لوحة الاستضافة كـ **Secrets / Environment variables** — لا تُرفع إلى GitHub).

---

## 3) مسارات واقعية (اختر ما يناسبك)

### أ) لديك ميزانية وتريد التركيز على الفكرة فقط

- **منصات تجارة جاهزة:** Shopify، Salla، Zid، WooCommerce على استضافة مدارة، إلخ.  
  تنفّذ الفكرة بسرعة دون صيانة كود Node/Vite.  
  **GitHub** يمكن أن يكون فقط لملفات التصميم أو التوثيق إن رغبت.

### ب) تريد الإبقاء على **هذا** المشروع (كود مفتوح + Supabase)

تحتاج **مستضيفاً يشغّل Node**، مثال شائع (أسماء للبحث، ليست إعلاناً):

- **Render**، **Railway**، **Fly.io**، **Coolify على VPS**  
  الفكرة: تربط المستودع من GitHub، تضبط **أمر البناء** و**أمر التشغيل**، وتلصق **نفس أسرار** Supabase و`DATABASE_URL` في لوحة المتغيرات.

**ملاحظة:** يحتاج من ينفّذ النشر (أنت بعد تعلّم أساسي، أو **مطوّر مستقل**) لأن كل منصة لها خطوات وواجهة مختلفة؛ لا يوجد زر واحد في GitHub ينشر هذا النوع من التطبيق كاملاً دون إعداد.

### ج) تريد **GitHub Pages** فقط (مجاني وبسيط)

ممكن **جزئياً:** عرض الواجهة الثابتة فقط من `dist/` **إذا** وُجدت طريقة لاستدعاء API من عنوان آخر (خادم منفصل) — هذا **يتطلب تعديلاً في الكود** (عنوان API ثابت أو متغير بيئة وقت البناء). **ليس جاهزاً «كما هو»** بدون خطوة برمجية.

---

## 4) ماذا تفعل عملياً كصاحب فكرة (غير مبرمج)؟

1. **اكتب فكرتك بوضوح:** من يبيع؟ ماذا يشتري الزبون؟ الدفع؟ التوصيل؟ اللغة؟  
2. **قرّر:** منصة جاهزة **أو** متابعة هذا المشروع مع مطوّر.  
3. إن اخترت هذا المشروع:  
   - ارفع الكود إلى **GitHub** (مستودع خاص إن أردت).  
   - عيّن **مطوّراً** أو تعلّم خطوة «ربط GitHub بـ Render/Railway» ونسخ **Secrets** من Supabase.  
4. **لا ترفع ملف `.env` الحقيقي** إلى GitHub — الأسرار تُضاف فقط في لوحة الاستضافة.

---

## 5) ربط مع باقي المستندات

- **`README.md`** — ما هو المشروع وتشغيله للتجربة (للمطور أو لك لاحقاً).  
- **`SUPABASE.md`** — قاعدة البيانات وSupabase CLI.  
- **`.env.example`** — قائمة المتغيرات التي سيضبطها المستضيف كـ Secrets عند النشر.

---

إن رغبت لاحقاً في **منصة محددة** (مثلاً Render فقط)، يمكن لمطوّرك اتباع وثائقها الرسمية مع: بناء الواجهة `npm run build`، وتشغيل الخادم `node server/index.js`، وتعيين `PORT` الذي توفّره المنصة.
